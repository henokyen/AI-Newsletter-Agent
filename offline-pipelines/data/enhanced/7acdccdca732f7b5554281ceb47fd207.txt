Graph Databases should be the better choice for Retrieval Augmented Generation (RAG)! RAG systems are usually pretty bad at retrieving the right context! The retrieved information is very local to the original documents, and it might be tricky to get the documents that actually relate to the user query. That is where Microsoft GraphRAG can provide a solution! There are 2 important pieces. First, we are going to create summaries of the different documents at different scales. This way, we are going to be able to understand, overall, what global information is contained in the full documents, as well as the local information contained in smaller chunks of text. Second, we are going to reduce the textual information to its graphical form. The assumption is that the information contained in the text can be represented as a set of nodes and edges. This allows the representation of the full information contained in the text as a knowledge base that can be stored in a graph database. The data indexing pipeline is as follows: 1) We chunk the original documents into sub-texts. 2) We extract the entities, their relationships, and their descriptions in a structured format by using an LLM. 3) We resolve the duplicate entities and relationships we can find across the different text chunks. We can summarize the different descriptions into more complete ones. 4) We build the graph representation of the entities and their relationships 5) From the graph, we cluster the different entities into communities in a hierarchical manner by using the Leiden algorithm. Each entity belongs to multiple clusters depending on the scale of the clusters. 6) For each community, we summarize it by using the entity and relationship descriptions. We have multiple summaries for each entity representing the different scales of the communities. At retrieval time, we can convert the user query by extracting keywords for a lexical search and a vector representation for a semantic search. From the search, we get entities, and from the entities, we get their related community summaries. Those summaries are used as context in the prompt at generation time to answer the user query. Graph Databases should be the better choice for Retrieval Augmented Generation (RAG)! RAG systems are usually pretty bad at retrieving the right context! The retrieved information is very local to the original documents, and it might be tricky to get the documents that actually relate to the user query. That is where Microsoft GraphRAG can provide a solution! There are 2 important pieces. First, we are going to create summaries of the different documents at different scales. This way, we are going to be able to understand, overall, what global information is contained in the full documents, as well as the local information contained in smaller chunks of text. Second, we are going to reduce the textual information to its graphical form. The assumption is that the information contained in the text can be represented as a set of nodes and edges. This allows the representation of the full information contained in the text as a knowledge base that can be stored in a graph database. The data indexing pipeline is as follows: 1) We chunk the original documents into sub-texts. 2) We extract the entities, their relationships, and their descriptions in a structured format by using an LLM. 3) We resolve the duplicate entities and relationships we can find across the different text chunks. We can summarize the different descriptions into more complete ones. 4) We build the graph representation of the entities and their relationships 5) From the graph, we cluster the different entities into communities in a hierarchical manner by using the Leiden algorithm. Each entity belongs to multiple clusters depending on the scale of the clusters. 6) For each community, we summarize it by using the entity and relationship descriptions. We have multiple summaries for each entity representing the different scales of the communities. At retrieval time, we can convert the user query by extracting keywords for a lexical search and a vector representation for a semantic search. From the search, we get entities, and from the entities, we get their related community summaries. Those summaries are used as context in the prompt at generation time to answer the user query. You can explore the code here. Introduction to RAG Retrieval-augmented generation (RAG) is currently one of the hottest topics in the world of LLM and AI applications. In short, RAG is a technique for grounding a generative models‚Äô response on chosen knowledge sources. It comprises two phases: retrieval and generation. In the retrieval phase, given a user‚Äôs query, we retrieve pieces of relevant information from a predefined knowledge source. Then, we insert the retrieved information into the prompt that is sent to an LLM, which (ideally) generates an answer to the user‚Äôs question based on the provided context. A commonly used approach to achieve efficient and accurate retrieval is through the usage of embeddings. In this approach, we preprocess users‚Äô data (let‚Äôs assume plain text for simplicity) by splitting the documents into chunks (such as pages, paragraphs, or sentences). We then use an embedding model to create a meaningful, numerical representation of these chunks, and store them in a vector database. Now, when a query comes in, we embed it as well and perform a similarity search using the vector database to retrieve the relevant information Image by the author If you are completely new to this concept, I‚Äôd recommend [deeplearning.ai](https://www.linkedin.com/redir/redirect?url=http%3A%2F%2Fdeeplearning%2Eai&urlhash=EbqT&trk=public_post-text) great course, LangChain: Chat with Your Data. What is ‚ÄúParent Document Retrieval‚Äù? ‚ÄúParent Document Retrieval‚Äù or ‚ÄúSentence Window Retrieval‚Äù as referred by others, is a common approach to enhance the performance of retrieval methods in RAG by providing the LLM with a broader context to consider. In essence, we divide the original documents into relatively small chunks, embed each one, and store them in a vector database. Using such small chunks (a sentence or a couple of sentences) helps the embedding models to better reflect their meaning [1]. Then, at retrieval time, we do not return the most similar chunk as found by the vector database only, but also its surrounding context (chunks) in the original document. That way, the LLM will have a broader context, which, in many cases, helps generate better answers. LangChain supports this concept via Parent Document Retriever [2]. The Parent Document Retriever allows you to: (1) retrieve the full document a specific chunk originated from, or (2) pre-define a larger ‚Äúparent‚Äù chunk, for each smaller chunk associated with that parent. Retrieval-Augmented Generation (RAG) Notes Basic Concepts and Components - Chunk: A single unit of data or information, often a part of a larger document. - Chunking (Text Splitting): Breaking down a long body of text into smaller parts for easier handling. - Document Loader: Imports documents into the retrieval system for indexing and retrieval. - Document Store (DocStore): A specialized database for managing and retrieving documents. - Document Pair Construction: Creating pairs of similar and dissimilar documents for training retrieval models. - Index: A data structure for fast retrieval of documents or chunks within a dataset, mapping key terms to their locations. Mathematical and Vector Representations - Embedding: A high-dimensional mathematical representation of a document or chunk. - Vector: Mathematical and Vector Representations - Dimension Size: The number of features in the space, corresponding to the size of embeddings. - Vector Store (VectorStore): A database for vectors, enabling efficient retrieval and comparison. Retrieval and Ranking Techniques - Retriever: Fetches relevant documents based on a query using embeddings and similarity measures. - Ranking Algorithms: Rank retrieved documents based on their relevance to the query. - Filtering Mechanisms: Refine search results based on criteria like date, location, or file type. - Relevance Feedback: Allows users to provide feedback on the relevance of retrieved documents to refine future search results. - Result Presentation: Displays retrieved information with snippets, summaries, and highlighted query terms. - Personalization: Tailors search results to individual users‚Äô preferences and interests. - Maximum Marginal Relevance (MMR): An algorithm providing relevant and diverse search results. Similarity and Relevance Measures - Cosine Similarity: Measures similarity between two vectors by calculating the cosine of the angle between them. - Term Frequency-Inverse Document Frequency (TF-IDF): Reflects the importance of a word in a document relative to a corpus. - Lexical Dependency Models: Consider dependencies between terms to improve retrieval effectiveness. - Sentiment: The emotional tone behind words, used to understand attitudes and emotions in text. Advanced Models and Techniques - Attention Mechanisms: Focus on specific parts of the input text to enhance retrieval accuracy. - Transformer Models: Advanced deep learning models like BERT that use self-attention to capture context and semantics. - Query Expansion: Improves retrieval by adding semantically related terms to the original query. - Hierarchical Document Representation: Represents documents hierarchically to encode and retrieve information at different levels. - Metric Learning: Measures similarity between documents using techniques like Siamese networks and triplet loss. Effective retrieval is a pain, and you can encounter several issues during this important stage. Here are some common pain points and possible solutions in the retrival stage. ‚Æï Challenge: Retrieved data not in context & there can be several reasons for this. ‚û§ Missed Top Rank Documents: The system sometimes doesn‚Äôt include essential documents that contain the answer in the top results returned by the system‚Äôs retrieval component. ‚û§ Incorrect Specificity: Responses may not provide precise information or adequately address the specific context of the user‚Äôs query ‚û§ Losing Relevant Context During Reranking: This occurs when documents containing the answer are retrieved from the database but fail to make it into the context for generating an answer. ‚Æï Proposed Solutions: ‚û§ Query Augmentation: Query augmentation enables RAG to retrieve information that is in context by enhancing the user queries with additional contextual details or modifying them to maximize relevancy. This involves improving the phrasing, adding company-specific context, and generating sub-questions that help contextualize and generate accurate responses - Rephrasing - Hypothetical document embeddings - Sub-queries ‚û§ Tweak retrieval strategies: Llama Index offers a range of retrieval strategies, from basic to advanced, to ensure accurate retrieval in RAG pipelines. By exploring these strategies, developers can improve the system‚Äôs ability to incorporate relevant information into the context for generating accurate responses. - Small-to-big sentence window retrieval, - recursive retrieval - semantic similarity scoring. ‚û§ Hyperparameter tuning for chunk size and similarity_top_k: This solution involves adjusting the parameters of the retrieval process in RAG models. More specifically, we can tune the parameters related to chunk size and similarity_top_k. The chunk_size parameter determines the size of the text chunks used for retrieval, while similarity_top_k controls the number of similar chunks retrieved. By experimenting with different values for these parameters, developers can find the optimal balance between computational efficiency and the quality of retrieved information. ‚û§ Reranking: Reranking retrieval results before they are sent to the language model has proven to improve RAG systems‚Äô performance significantly. This reranking process can be implemented by incorporating the reranker as a postprocessor in the RAG pipeline.
